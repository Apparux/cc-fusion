# Design: Statusline Tasks and Context Stability

## Scope

Fix two statusline data-quality issues without changing the overall layout, theme, preset model, or Claude Code upstream behavior:

1. Tasks should reflect parseable Claude Code task state even when task creation happened earlier than the normal transcript tail window.
2. Context should not render missing usage data as a real `0.0% / 0` reading.

## Boundaries

- Keep Claude Code stdin compatibility in `src/stdin.ts`.
- Keep transcript and task aggregation in `src/transcript.ts`.
- Keep user-visible terminal rendering in `src/lines/line2.ts` and `src/lines/line4.ts` only when formatting needs to change.
- Do not add runtime dependencies or debug logging.
- Do not hand-edit `dist/`; build output is generated by `npm run build`.

## Context Data Contract

Current context calculations collapse missing values to `0`, which makes unknown usage indistinguishable from real zero usage. Add an explicit availability signal from the stdin compatibility layer so renderers can distinguish:

- Known percentage: render current percentage, progress bar, and token values as today.
- Unknown percentage/usage with known window size: render unknown percentage and used tokens with known total, e.g. `--.-%  -- / 1.0M tokens`.
- Unknown percentage/usage with unknown window size: render unknown placeholders for both used and total values.

The implementation should not cache previous invocations. Each statusline render remains stateless and based only on current stdin.

## Tasks Data Contract

Current task aggregation needs both task identity and subject. Claude Code commonly emits `TaskCreate` without `taskId`, then emits a `tool_result` like `Task #N created successfully: <subject>`. For long transcripts, the default tail window can contain `TaskUpdate` entries whose corresponding create/result entries are outside the tail window.

Task aggregation should therefore reconstruct the current task batch from a broader task-event scan while keeping other activity fields bounded:

- Parse task create/result/update events in transcript order.
- Map `TaskCreate` tool IDs to subjects.
- Use creation result text to assign task IDs when tool input lacks `taskId`.
- Apply `TaskUpdate` to existing tasks once IDs are known.
- Preserve existing batch semantics: when all currently tracked tasks are done, the next distinct created task starts a fresh batch; unfinished batches append later tasks.
- Preserve display-window semantics: calculate totals from the full current batch, but render at most the first five non-done tasks when there are more than five.

For activity fields such as last read/edit/search and agents, keep the existing tail-bounded behavior so statusline execution stays cheap.

## Compatibility and Trade-offs

- Existing transcripts with `TaskCreate.input.taskId` must keep working.
- Existing task-result dedup behavior must keep working.
- Broader scanning increases local file reads for task events, but avoids the observed false “无待办任务” when create events are older than the tail window.
- Context unknown placeholders are more honest than `0.0%`, but create a mixed known/unknown display when only context window size is available.

## Rollback

The change is localized to stdin compatibility, transcript parsing, and renderer formatting. Rollback is a normal git revert of this task's commit. If performance regressions appear, reduce the task-event scan bound while retaining a regression test for tail-window mismatch.
